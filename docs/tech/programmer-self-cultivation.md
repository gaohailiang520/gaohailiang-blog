## 第二部分：静态链接

### 第 2 章：编译和链接

> **核心问题**：从 `hello.c` 到可执行文件，编译器、汇编器、链接器分别做了什么？

```
源代码 (hello.c)
      ↓ [预处理] hello.i (展开宏)
      ↓ [编译] hello.s (汇编语言)
      ↓ [汇编] hello.o (目标文件/机器码)
      ↓ [链接] hello (可执行文件)
```

**关键概念**：

| 阶段 | 工具 | 输出 | 主要任务 |
|------|------|------|---------|
| 预处理 | gcc -E | .i 文件 | 展开宏、删除注释 |
| 编译 | gcc -S | .s 文件 | 语法分析、生成 AST、生成汇编 |
| 汇编 | as / gcc -c | .o 文件 | 汇编语言 → 机器码 |
| 链接 | ld | 可执行文件 | 符号解析、重定位、合并段 |

> **驱动开发意义**：理解编译过程才能理解 Makefile、交叉编译、编译器优化选项（如 `-O2`、`-fno-stack-protector`）

### 第 3 章：目标文件里有什么

> **核心问题**：目标文件（.o）里到底有什么？

**目标文件格式**：

| 格式 | 操作系统 |
|------|---------|
| ELF | Linux |
| PE/COFF | Windows |
| Mach-O | macOS |

**ELF 文件结构**：

```
┌────────────────────────────┐
│ ELF Header                 │  ← 文件基本信息、入口地址
├────────────────────────────┤
│ Program Header Table       │  ← 段表（装载时用）
├────────────────────────────┤
│ Section 1: .text           │  ← 代码段
│ Section 2: .data           │  ← 已初始化数据
│ Section 3: .bss            │  ← 未初始化数据
│ Section 4: .rodata         │  ← 只读数据
│ Section 5: .symtab         │  ← 符号表
│ Section 6: .rel/.rela      │  ← 重定位表
│ Section 7: .debug          │  ← 调试信息
│ ...                        │
├────────────────────────────┤
│ Section Header Table       │  ← 节表（链接时用）
└────────────────────────────┘
```

**关键段（Section）**：

| 段名 | 内容 | 用途 |
|------|------|------|
| `.text` | 机器码 | 代码 |
| `.data` | 已初始化全局/静态变量 | 数据 |
| `.bss` | 未初始化全局/静态变量 | 数据（不占文件空间） |
| `.rodata` | 只读常量（字符串、const） | 数据 |
| `.symtab` | 符号表 | 链接时符号解析 |
| `.rel/.rela` | 重定位信息 | 链接时地址修正 |

> **驱动开发意义**：理解 ELF 结构才能理解内核模块（.ko）、符号导出/导入、内核符号表

### 第 4 章：静态链接

> **核心问题**：链接器如何把多个 .o 文件合并成一个可执行文件？

**两步过程**：

1. **空间分配**
   - 按序叠加：简单但浪费空间
   - 合并相似段：所有 .text 合并为一个 .text，所有 .data 合并为一个 .data（**主流方式**）

2. **符号解析与重定位**
   - 符号定义（Symbol Definition）vs 符号引用（Symbol Reference）
   - 链接器在所有目标文件中查找符号定义，找不到则报"未定义符号"错误
   - 重定位：把符号的相对地址修正为绝对地址

**强符号 vs 弱符号**：

```c
// a.c
int global = 1;        // 强符号
int weak;             // 弱符号

// b.c  
int global;            // 强符号优先级高，覆盖 a.c 的 global

// c.c
int global;            // 弱符号
float global;         // 冲突！链接器报错
```

## 第三部分：装载与动态链接

### 第 6 章：可执行文件的装载与进程

> **核心问题**：操作系统如何把可执行文件加载到内存并创建一个进程？

**进程虚拟地址空间**：

```
高地址 ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ → 低地址
┌─────────────────────────────┐
│        内核空间              │  ← 只有内核能访问
├─────────────────────────────┤
│        用户空间              │
│  ┌───────────────────────┐  │
│  │ 栈 (Stack)            │  │  ← 向低地址增长
│  │        ↓              │  │
│  │                       │  │
│  │        ↑              │  │
│  │  堆 (Heap)            │  │  ← 向高地址增长
│  │  .bss                 │  │
│  │  .data                │  │
│  │  .text                │  │
│  │  映射区域              │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
```

**装载方式**：

| 方式 | 描述 | 特点 |
|------|------|------|
| 覆盖装载 | 早期 DOS，内存不够时手动切换模块 | 古老、复杂 |
| 动态装载 | 现代做法，操作系统管理虚拟内存 | 由 Loader 完成 |

**Linux 装载 ELF 流程**：

1. 内核读取 ELF Header
2. 读取 Program Header，确定需要映射的段
3. 建立虚拟地址空间映射（页表）
4. 将可执行文件的段映射到内存
5. 设置 CPU 指令寄存器指向入口地址
6. 跳转到入口，开始执行

> **驱动开发意义**：理解进程地址空间布局，才能理解内核态与用户态的划分、驱动如何与用户态程序共享内存

### 第 7 章：动态链接

> **核心问题**：为什么需要动态链接？它是如何工作的？

**静态链接的问题**：

```
┌─────────────────────────────────────────┐
│  静态链接的可执行文件 A                  │
│  ┌────────┐ ┌────────┐ ┌────────┐        │
│  │ 代码段  │ │ lib.a │ │ libc.a │        │  ← 每个程序都复制一份
│  └────────┘ └────────┘ └────────┘        │
└─────────────────────────────────────────┘

问题：
1. 浪费磁盘空间（每个可执行文件都带一份库）
2. 浪费内存（每个进程都加载一份库代码）
3. 更新库需要重新链接所有程序
```

**动态链接的解决方案**：

```
┌─────────────────┐     ┌─────────────────┐
│  可执行文件 A   │     │  共享库 libc.so │
│  ┌───────────┐  │     │  ┌────────────┐  │
│  │ 代码段    │  │ ──→ │  │  代码段    │  │  ← 多个进程共享一份
│  └───────────┘  │     │  └────────────┘  │
│  ┌───────────┐  │     └─────────────────┘
│  │ .got/.plt │  │     
│  └───────────┘  │     内存中：
└─────────────────┘     ┌───────────────┐
                       │ 进程 A │ libc │
                       │ 进程 B │ libc │  ← 共享
                       └───────────────┘
```

**地址无关代码（PIC, Position Independent Code）**：

- 共享库代码可以加载到任意地址
- 关键：**相对地址**而非绝对地址
- 编译时添加 `-fPIC` 参数

**延迟绑定（Lazy Binding）**：

```
// 第一次调用 printf
1. 跳转到 PLT[printf]
2. PLT 跳转到链接器
3. 链接器解析 printf 地址，写入 GOT
4. 以后直接跳到 GOT[printf]（已解析）
```

> **驱动开发意义**：理解动态链接是理解内核模块（.ko）加载、内核符号表、/proc/kallsyms 的基础

### 第 8 章：Linux 共享库的组织

**SO 命名规则**：

```
libname.so.x.y.z
  │     │ │ └── 版本（发布版本）
  │     │ └────── 接口版本（增量升级，兼容）
  └─────┴───────── 主版本号（不兼容更改）
```

**系统路径**：

```
/lib/              → 关键共享库
/lib64/            → 64 位库
/usr/lib/          → 标准库
/usr/local/lib/    → 第三方库
```

**环境变量**：

| 变量 | 作用 |
|------|------|
| `LD_LIBRARY_PATH` | 临时添加库搜索路径 |
| `LD_PRELOAD` | 强制加载指定库（可用来 hook 函数） |
| `LD_DEBUG` | 调试模式，显示动态链接过程 |

> **驱动开发意义**：`LD_PRELOAD` 是理解 eBPF、ftrace 等内核级追踪技术的用户态基础

## 第四部分：库与运行库

### 第 10 章：内存

**栈（Stack）**：

```
高地址 ← ─ ─ ─ ─ ─ ─ → 低地址
┌────────────────────────┐
│  返回地址               │  ← 函数返回后跳转的地址
├────────────────────────┤
│  保存的寄存器           │  ← callee-saved registers
├────────────────────────┤
│  局部变量               │  ← 本地变量
├────────────────────────┤
│  函数参数               │  ← 参数（从右向左入栈）
└────────────────────────┘
          ↓ 函数调用
          ↑ 函数返回
```

**调用惯例（Calling Convention）**：

| 惯例 | 参数传递 | 栈清理 | 使用平台 |
|------|---------|-------|---------|
| cdecl | 栈（从右向左） | 调用者 | x86 GCC |
| stdcall | 栈（从右向左） | 被调用者 | x86 Windows |
| fastcall | 寄存器 + 栈 | 被调用者 | x86 |
| syscall | 寄存器 | 调用者 | Linux x86_64 |

**堆（Heap）分配算法**：

| 算法 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| 首次适应 | 找第一个足够大的空闲块 | 简单 | 碎片化 |
| 最佳适应 | 找最小的足够空闲块 | 内存利用率高 | 慢 |
| 快速分配 | 预分配固定大小块池 | 极快 | 内存浪费 |

> **驱动开发意义**：理解栈的布局和函数调用约定，才能正确编写内核回调函数、处理中断上下文

### 第 11 章：运行库

**C 运行库做了什么**：

```
main() ← 程序入口
  ↓
__start → __libc_start_main → main
         ↓
    ┌────┴────┐
    ↓         ↓
  初始化    清理
  - 初始化全局变量
  - 设置栈
  - atexit() 注册
    - 静态对象析构
    - fflush()
```

**关键函数**：

| 函数 | 作用 |
|------|------|
| `main()` | 程序入口（其实是用户代码入口） |
| `_start()` | 真正的入口，设置环境后调用 main |
| `exit()` | 正常退出，调用 atexit 回调 |
| `_exit()` | 直接终止进程（不刷新缓冲区） |

### 第 12 章：系统调用与 API

**系统调用原理**：

```
用户态                          内核态
┌─────────────┐                ┌─────────────┐
│  printf()   │                │  sys_write()│
│  (libc)     │ ──→ 中断 ──→   │  (kernel)   │
└─────────────┘    0x80        └─────────────┘
```

**x86 系统调用方式**：

| 架构 | 中断/指令 | 例子 |
|------|----------|------|
| x86 | int 0x80 | Linux 32-bit |
| x86_64 | syscall | Linux 64-bit |
| ARM | svc 0 | Linux ARM |

**常见系统调用**：

| 系统调用 | 功能 | 驱动中的对应 |
|----------|------|-------------|
| `read()` | 读文件/设备 | file_operations.read |
| `write()` | 写文件/设备 | file_operations.write |
| `open()` | 打开文件/设备 | file_operations.open |
| `ioctl()` | 控制设备 | file_operations.unlocked_ioctl |
| `mmap()` | 映射内存 | file_operations.mmap |
| `poll()` | 等待事件 | file_operations.poll |

## 总结

### 核心概念地图

```
源代码 (hello.c)
      ↓
预处理 + 编译 + 汇编
      ↓
目标文件 (hello.o) → ELF 格式
      ↓
链接器 + 符号解析 + 重定位
      ↓
可执行文件 (hello)
      ↓
Loader 装载 → 进程虚拟地址空间
      ↓
运行时 → 动态链接（.so）
      ↓
内存布局：代码段、数据段、堆、栈
      ↓
系统调用 → 内核
```

### 驱动开发者必知

1. **编译链接**：会写 Makefile，理解 `-fPIC`、`-shared`
2. **ELF 结构**：能分析 .ko 文件的段和符号
3. **动态链接**：理解 PLT/GOT、延迟绑定
4. **地址空间**：理解用户态/内核态隔离
5. **系统调用**：理解 write/read/ioctl 在内核中的路径
6. **内存管理**：理解 kmalloc、vmalloc、页分配的区别
